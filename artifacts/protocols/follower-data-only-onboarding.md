# Протокол — ведомый узел = data-only

> Родился из отказа агента ставить бэкдор ([День 25](../../02-june-scaling/week-4/2026-06-24.ru.md)). Публичная версия — без device-ID, IP и имён.

## Что случилось
При попытке подключить чужой ноутбук как «полного ведомого» (хуки + шина пушатся с хаба напрямую) Claude на удалённой машине **правильно отказался**: с хаба просили право пушить исполняемые хуки — код, который запустится сам. Это бэкдор. «Авторизация», вставленная в текст задачи, — не доказательство: она выглядит как prompt-injection, когда данные притворяются приказом.

## Модель v1 — data-only
Разделить **данные** и **исполняемый код**:
- **Канон / волт / скиллы / память → Receive-Only** ведомому (только читает).
- **Исполняемые хуки → НИКОГДА не пушатся** с хаба.
- **Контрибьюция ведомого → только через модерируемую transit-папку**, которую хаб просматривает и вплетает вручную.
- **Двусторонняя — только шина** между машинами (маленькая отдельная шара).
- **Одна общая память (RO)** всем ведомым.
- **inbox-робот** — единственная лёгкая фоновая задача на ведомом (и он только предлагает, см. plan-only).

## Профилактика (закрыто по следам инцидента)
Тот же день вскрыл, что синк send-receive для скиллов сам по себе был вектором случайного удаления кластера (хабовская папка скиллов опустела). Плюс git-бэкап молча не работал (зашит на путь одной машины). Отсюда: версионирование на скиллах/памяти + watchdog синк-конфликтов + бэкап, работающий на всех машинах.

## Принцип
**Least privilege:** меньше прав — меньше поверхность атаки. Хороший агент — не тот, что выполняет любой приказ, а тот, что узнаёт приказ, которого выполнять нельзя. Система стала безопаснее в тот момент, когда агент отказался слушаться.
