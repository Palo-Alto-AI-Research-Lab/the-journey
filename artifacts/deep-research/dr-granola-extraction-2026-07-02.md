## Contradictions & Debates

The biggest contradiction is between **what Granola officially documents in detail** and **what it alludes to in marketing/admin material**. The public docs fully document the **personal/public notes API** for Business and Enterprise users, but the formal public reference does **not** document the separate **Enterprise API** that GranolaŌĆÖs March 2026 blog announced for admins working with team context. Meanwhile, the Enterprise admin settings page says **admin-level data export tools are separate** and remain available regardless of user export toggles. The practical reading is: **Enterprise-only admin-level export capability likely exists, but its technical surface is either private, sales-led, or not publicly documented yet**. That is important, because if your organization can get access to that admin path, it may supersede some of the workarounds below. ŅłĆciteŅłéturn36view0Ņłéturn37view0Ņłéturn35search8Ņłéturn14view0Ņłü

The second contradiction is around **local auth/token storage**. Older 2025 reverse-engineering and blog posts point to `~/Library/Application Support/Granola/supabase.json` holding WorkOS tokens, and some newer tools still rely on that file. But more recent open-source plugins describe a newer model with `stored-accounts.json` or `stored-accounts.json.enc`, with the wrapping key protected by macOS Keychain, Linux secret stores, or **Windows DPAPI** in `%APPDATA%\Granola\Local State`. GranolaŌĆÖs own April 2026 changelog, which announced ongoing **local cache encryption**, strongly supports the view that storage internals have been actively changing. In plain English: **if you build against on-disk auth material, you are building against a moving target**. The third contradiction is around **cache versions and semantics**. The historical community reference point was `cache-v3.json` on macOS. A later open-source plugin documented a February 2026 breakage to **cache-v4**, along with structural changes and the claim that AI summaries moved to the cloud. A newer sync tool in 2026 now auto-detects **cache-v6** and assumes the highest `cache-vN.json` file wins. ## Best Practices

For extraction, the best current pattern is **official API first, folder-based metadata second, CSV only as a backup, and private/cache methods only as contingency tooling**. The public API is supported, documented, rate-limited, and aligned with GranolaŌĆÖs own direction of travel. Cache and private-API methods are now explicitly discouraged by GranolaŌĆÖs product team, and the change cadence in the cache formats proves the reason why. ŅłĆciteŅłéturn14view0Ņłéturn13view0Ņłéturn12search5Ņłü

For your **one-time full backfill**, build a bounded crawler around the official API. Enumerate folders with cursor pagination, enumerate notes, and persist immutable upstream identifiers plus sync cursors in SQLite. Use the CSV export once as a **reconciliation artifact** rather than as the import source: it is useful for comparing counts and for recovering edge notes that were somehow missed, but its ŌĆ£summary-only / owned-only / once-per-day / link-expiresŌĆØ constraints are too awkward for primary ingestion. ŅłĆciteŅłéturn7view2Ņłéturn14view0Ņłéturn9view0Ņłü

For the **nightly unattended incremental**, prefer one of two patterns. The first is simple polling against the official API using API keys, respecting the 5 rps ceiling and using `updated_at`-style watermarks or per-note change detection from periodic note re-fetches. The second is more event-driven: create a Granola folder policy and push notes to **Zapier** as soon as they enter that folder, then forward the Zapier payload to your local automation endpoint or durable queue. That second pattern is the closest you can get to ŌĆ£webhooksŌĆØ in Granola today without a documented public webhook API. ŅłĆciteŅłéturn14view0Ņłéturn9view1Ņłéturn12search1Ņłéturn9view2Ņłü

For note routing, exploit GranolaŌĆÖs **folder auto-add for recurring meetings** and **folder auto-post** features. If you add one meeting in a recurring series to a folder and enable auto-add, future meetings in that series can be routed into the same folder automatically; once there, folder integrations can auto-post them to Slack, HubSpot, Attio, or Zapier. The practical consequence is that Granola does have a workable **semantic routing plane**; the missing piece is a general first-party webhook endpoint. ŅłĆciteŅłéturn10search2Ņłéturn9view2Ņłéturn9view3Ņłéturn9view4Ņłü

For local durability, standardize on **derived artifacts**, not raw Granola internals. Store raw API responses, normalized markdown, a SQLite index, and your own per-meeting hashes. That way, if Granola changes markdown structure, folders, or transcript formatting, you only need to update your parser layer. Open-source Obsidian plugins that survived format churn all converged on the same patterns: **meeting-ID-based deduplication**, **periodic sync**, **skip-existing / preserve-local-edits options**, and **auto-detecting format changes rather than hardcoding one file shape**. For coverage, operational best practice is more conservative: **one active capture device per meeting**. GranolaŌĆÖs docs do not define conflict semantics for parallel desktop capture, but they do define capture as local to the device audio path. ## Failure Modes

The most common product-level failure is the simple one: **the meeting was never actually captured**. GranolaŌĆÖs own docs repeat this in multiple ways. Notifications can appear, the app can be open, the icon can sit in the tray, and yet nothing is recorded unless the user opened the meeting note or started a Quick Note. If your team assumes ŌĆ£app running = recording,ŌĆØ your pipeline will be cleanly engineered and still miss meetings. ŅłĆciteŅłéturn9view6Ņłéturn11view2Ņłéturn11view1Ņłü

The second failure mode is **OAuth/session fragility in MCP-based automations**. Granola MCP itself is browser-OAuth-only, and external client issues in 2026 show expiring tokens, silent failures, and missing refresh behavior. That matches your observed ŌĆ£silently expired after ~10 daysŌĆØ experience closely enough that it should be treated as a category problem, not a one-off incident. ŅłĆciteŅłéturn15search14Ņłéturn31view1Ņłéturn31view2Ņłéturn31view3Ņłü

The third failure mode is **depending on private endpoints or local cache contracts that Granola is actively mutating**. The reverse-engineered repo that once documented private endpoints is now archived specifically because Granola released an official API. Multiple open-source tools broke when the cache changed, including the v3ŌåÆv4 jump; one current local-cache MCP server explicitly warns that GranolaŌĆÖs cache format may change without notice; and GranolaŌĆÖs own changelog states that cache/private API workflows were never a supported target. The fourth failure mode is **assuming integrations are retroactive**. Folder-based auto-post to Slack, HubSpot, and Zapier applies only to notes added **after** the automation is enabled. ZapierŌĆÖs folder trigger also only fires for future additions. If you switch on automation after months of meetings already exist, it does nothing for the historical corpus. The fifth failure mode is **Windows/macOS credential-path assumptions hardcoded from one era**. ## Strategic Recommendations

The core recommendation is blunt: **build the production extractor on the official API now, and treat everything else as either a fallback or a migration bridge**. In early 2025, reverse engineering was rational because Granola lacked a public API. In mid-2026, it is no longer rational as the default. GranolaŌĆÖs product direction, docs, and changelog all point the same way. ŅłĆciteŅłéturn14view0Ņłéturn12search5Ņłéturn13view0Ņłü

For your specific architecture, I would rank paths like this:

**Best production path:** official API with API keys on Business/Enterprise.  
**Best push-style augmentation:** folder routing + Zapier trigger forwarding transcript payloads.  
**Acceptable emergency fallback:** daily/weekly CSV export for reconciliation or disaster recovery.  
**Interactive analyst path only:** MCP.  
**Last-resort break-glass path:** local cache / private endpoints, isolated behind feature flags and expected to break. ŅłĆciteŅłéturn12search0Ņłéturn9view1Ņłéturn9view0Ņłéturn15search14Ņłéturn13view0Ņłü

## Action Plan

**Immediate move:** create a dedicated Granola API key with the narrowest scopes that still satisfy your archive requirement, then implement the official-API backfill first. Use a local schema with tables for workspaces, folders, notes, transcripts, sync checkpoints, and raw payload snapshots. Treat meeting IDs as immutable keys and keep markdown generation as a pure downstream transform. This gets you out of the token-expiry trap immediately. ŅłĆciteŅłéturn12search0Ņłéturn14view0Ņłü

**Next move:** add a nightly incremental sync that runs from API keys only. Respect the 5 rps cap, keep exponential backoff for 429/5xx responses, and do periodic re-fetches of recently updated notes rather than assuming notes are immutable right after creation. Granola notes can continue to evolve after the meeting due to enhancement, folder movement, and user edits. The official docs do not explicitly define webhook semantics or update guarantees, so the safe engineering stance is incremental reconciliation rather than single-pass ingestion. ŅłĆciteŅłéturn14view0Ņłéturn7view2Ņłü

**Parallel move:** set up Granola folders as routing units for your call categories, enable **auto-add for recurring meetings**, and connect at least one **Zapier folder trigger** that forwards the transcript-bearing payload to your own ingestion endpoint. This gives you a pseudo-push channel for new notes, reduces polling pressure, and provides early warning if the polling job and the folder workflow disagree. ŅłĆciteŅłéturn10search2Ņłéturn9view1Ņłéturn9view2Ņłü

**Coverage hardening:** on each possible capture machine, use OS-native startup/login-item controls plus a process health monitor to ensure Granola is launched and updated. On Windows, Granola is a per-user app that can auto-update when installed in a user-writable location; Granola also publishes stable installer/version endpoints that your IT tooling can poll. **Break-glass fallback:** if official API coverage is temporarily insufficient for a subset of notes, keep a quarantined fallback toolchain that can inspect local cache/auth state on a single sacrificial machine.
