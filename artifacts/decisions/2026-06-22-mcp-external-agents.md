# Decision Memo — MCP над CRM для сторонних AI-агентов

> Триггер: звонок с реальным клиентом породил идею — дать ЧУЖИМ AI-агентам доступ к нашим данным/аутричу через MCP-сервер, «AI ↔ AI». Alpha Protocol: recall (идея всплывала ещё осенью) → 2 внешних DR (безопасность + рынок) → синтез. (Публичная версия — клиент и суммы опущены.)

## Проблема
Мир переходит от UX к **AX (Agent Experience)** — у каждой компании будет свой агент. Как дать стороннему AI-агенту (например, Codex клиента) обращаться к нашей CRM/аутричу напрямую, без человеческого интерфейса, — но так, чтобы это было безопасно и на нашей стороне?

## Deep Research — два трека сошлись в одной архитектуре
- **Брокер запросов ≠ исполнитель действия.** Внешний агент может только READ + REQUEST. Исполняет действие ВСЕГДА аккаунт человека-владельца, под его апрувом (human-in-the-loop). Это же совпало с давним собственным правилом «свой аккаунт, не боты» — введённым по другой причине (прозрачность): безопасность и старая привычка сошлись сами.
- **Массовые инвайты убьют аккаунты** (жалобы → баны). Поэтому разворот: не «затолкать людей», а **привратник наоборот** — сделать так, чтобы люди сами подавали join-request (pull, не push).
- **Настоящий ров — не софт, а поверхность распространения:** десятки тысяч тёплых контактов + право на тёплое интро. HubSpot/Salesforce этого не дают. Софт перепишут за неделю; доверие людей — нет.

## Решение
1. Архитектура: **брокер (read + request) на нашей стороне; execute — только свой аккаунт под апрувом.**
2. Прототип Phase 1 = read-only + «привратник наоборот» (pull join-request).
3. Монетизация: гибрид SaaS + usage/outcome-метеринг первым; токен — опциональный слой, НЕ ворота доступа.
4. Начинать с Telegram; WhatsApp-исполнение отложить.

## Стандарты, на которые опирались
MCP, Google A2A, AP2 (Agent Payments Protocol), x402 (Coinbase), OWASP MCP Security. Публичные цифры agent-initiated экономики (Alipay, Intercom Fin, Expedia, Microsoft Copilot Checkout) подтвердили, что рынок AI↔AI-транзакций уже реален.

## Открытый пункт
Билд Phase 1 (прототип для конкретного клиента) ждёт финального дизайна. Архитектурная логика («брокер ≠ исполнитель», moat = distribution surface) зафиксирована.
