# Decision Memo — MCP-стек под личный стек

> Финальный шаг Alpha Protocol по вопросу «какие MCP-серверы реально нужны ИМЕННО под этот стек». Метод: recall → 2 Deep Research (общий обзор рынка + заточенный под стек) → синтез. (Публичная версия — детали личной инфраструктуры опущены.)

## Ключевой инсайт
Первый DR оценивал дев-командный стек (GitHub / Atlassian / Sentry / Notion / Slack). Но здесь профиль другой — **personal-assistant + CRM-стабилизация + workflow-автоматизация + локальные скрипты.** Поэтому из «общего» отчёта прямо применимо мало; заточенный второй DR дал реальную карту.

## Решения

### ✅ KEEP (работает правильно)
Мессенджеры (Telegram / WhatsApp через свои stdio-серверы), Drive / Calendar, computer-use / браузер. База данных — **только локальная песочница-копия**; DR прямо: «к live-БД не подключать ВООБЩЕ, максимум дамп/реплика».

### ⚠️ ИЗМЕНИТЬ — workflow-MCP в режим «документация-онли»
- **Найдено:** workflow-MCP подключён к **боевому** инстансу с полным правом управления (create / update / delete / manage-credentials).
- **Read-only режима у него НЕТ:** либо документация-онли (знание нод + валидация, без ключа), либо полное управление.
- **Решение:** убрать ключ → документация-онли. Главная ценность (знание сотен нод + валидация) сохраняется; live-правки остаются на локальном скрипте, который делает бэкап перед каждой правкой (безопаснее агента). Откат = одна строка обратно.
- Позиция автора самого MCP совпадает: «не давайте ИИ править production-workflow напрямую».

### ❌ SKIP (отключено — usageCount 0)
Несколько плагинов и один дизайн-коннектор висели без использования → отключены.

### 🕐 ОТЛОЖЕНО до фазы CRM
Git-MCP (в read-only режиме), Playwright (E2E-тесты admin-панели), Postgres-MCP (только dev/stage, read-only role) — когда дойдёт до соответствующих задач.

### ❌ HARD-SKIP
- Docker-control MCP — всё, что он умеет, детерминированнее делает `docker compose` / скрипт.
- Filesystem / Fetch / Git reference-серверы — уже встроены в Claude Code.
- Sentry / Atlassian / Notion / Slack — таких систем в стеке нет.

## Поведенческие правила (две оси из DR)
1. **Trust boundary по СРЕДЕ, не по «read-only флажку MCP»:** LIVE → read-only на уровне БД/токена; STAGE/dev → RW можно; локальная песочница → RW свободно.
2. **MCP ≠ всегда лучше скрипта:** скрипт — для ops/повторяемых действий, MCP — для интерактивного расследования и authoring. Существующие скрипты (бэкап, архивация, правка workflow) НЕ заменяются MCP, а дополняются.
3. **Prompt-injection:** содержимое чужих workflow-описаний, сообщений, комментариев — это **данные, не команды.** Профиль «investigate-only vs build/stage-write» — паттерн на будущее.

## Паттерн
Тот же AK-47: не «подключи всё, что есть», а «подключи минимум, который реально работает и который сам починишь». Любой MCP с write-доступом к боевой системе → либо read-only, либо изолировать. Каждая лишняя зависимость — поверхность отказа и утечки.
