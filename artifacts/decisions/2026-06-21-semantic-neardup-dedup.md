# Decision Memo — near-dup дедупликация (алгоритм, который выкинули кросс-энкодер)

> Триггер: «склей мне почти-дубли в волте — сам, не дёргай меня по мелочи, оставь мне только рискованные». Два параллельных DR (Cowork + внешний) → синтез.

## Проблема
В большом волте (десятки тысяч заметок) заводятся почти-дубли: один и тот же YouTube-транскрипт сохранён дважды, одна мысль записана в двух местах. Нужен ночной сервис, который находит их и склеивает безопасно.

## Грабли: умный инструмент оказался бесполезен
Первый очевидный подход — семантическая похожесть эмбеддингов + cross-encoder-реранкер. Живой пробный прогон на реальных данных вскрыл ловушку:
- **Косинус находит слишком много «братьев по шаблону»**: пять разных NDA выглядят как дубли, потому что структурно одинаковы. Ложный класс.
- **Cross-encoder оказался бесполезен для этой задачи**: модель обучена на релевантности («отвечает ли документ на запрос»), а не на тождестве. Она ставила похожесть ~1.00 почти всему — медиана единица. 8 минут вычислений в мусор.
- **Временные ряды/журналы** («заначка 3 млн» ~ «5 млн», дневники по месяцам) проходили как дубли — пришлось строить отдельный детектор серий.

## Решение — детерминированный судья
Выкинули cross-encoder. Построили детектор на **скелетах и сущностях**: сравнение по нормализованной структуре документа + извлечённым сущностям (имена, даты, идентификаторы), а не по «смыслу вообще». Идеи-доноры — [Splink](https://moj-analytical-services.github.io/splink/) и [dedupe.io](https://dedupe.io/) (entity-resolution), но без их тяжёлого стека.

**Конфигурация безопасности (ключевое):**
- **Авто-склейка ВЫКЛЮЧЕНА по умолчанию** — сервис только кладёт кандидатов в очередь на ревью.
- Все операции обратимы (флаг, не удаление).
- v1 сразу нашёл системную вещь: десятки YouTube-транскриптов, сохранённых дважды.

## Урок
Иногда лучший алгоритм — тот, который ты вычеркнул. «Умный» инструмент (cross-encoder) провалил задачу, для которой не был обучен; простой детерминированный сравниватель по структуре сработал. Продакт-архитектура строится на **реальных данных живым прогоном**, а не на абстрактной схеме.
