# Decision Memo — единый триггер-слой (signal-dispatcher)

> Принято через Alpha Protocol (R+DR) 2026-06-21/22: два внешних DR-отчёта + Cowork-разбор на нашем Втором Мозге + собственная capability-верификация по докам Anthropic — **четыре независимых источника, единогласны.**

## Проблема
Claude Code — процесс **по требованию, не демон**: он работает, только пока открыта сессия. Событие случилось (ресёрч добит, файл приехал, upstream-джоба легла, настал дедлайн) — и **никто не толкает работу дальше.** Если лечить это отдельным сторожем под каждый случай → зоопарк почти одинаковых watcher'ов, у каждого свой лог, свои грабли, свой bus-factor. Нужен ОДИН общий механизм, чинибельный не-инженером (AK-47).

## Что знаем (Recall)
Уже построено: планировщик → headless-прогон, cron через MCP, always-on демон, детерминированные 0-token сторожа → пинг, SessionStart-хуки, межмашинная шина. Известная мина: второй MCP-клиент на одной Telegram-сессии = `AUTH_KEY_DUPLICATED` (вылет).

## Deep Research (4 источника сошлись)
Единый вывод: **один mailbox + один диспетчер**, ось `ping человеку` vs `self-wake`. Cowork добавил ключевое AK-47-уточнение: не строить новую БД-очередь с нуля, а **обобщить уже работающий file-mailbox + ledger.** Собственная верификация подтвердила все несущие факты (детерминированный session-id, budget-потолки, permission-mode, что облачные рутины НЕ видят локальный RAG/SQLite/личный Telegram).

## Варианты
- **A. Оставить зоопарк bespoke-сторожей** — отклонён: sprawl, bus-factor, тихие поломки.
- **B. Новая SQLite-очередь + диспетчер с нуля** — отложен: мощно, но новый код там, где есть рабочий аналог; менее AK-47. Порог возврата — когда источников станут десятки И появится многошаговый stateful workflow.
- **C. ✅ ПРИНЯТО: обобщить существующий watch-цикл → `signal-dispatcher` + `signals.yaml`**, file-mailbox + существующий ledger.
- **D. Только облачные рутины** — частично: годится лишь для чисто-промптовых cloud-safe задач (облако слепо к localhost).

## Принято (вариант C)
1. **Один table-driven диспетчер**, запускаемый ОДНОЙ записью планировщика каждые N минут («run asap after a missed start»).
2. **Источники только КЛАДУТ сигнал** (файл-маячок, mtime папки, дроп `.json`, расписание в yaml). Диспетчер — единственный, кто роутит.
3. **Default route = `ping` человека** (0 токенов). `wake` (headless-прогон) — только по явному флагу на конкретном сигнале.
4. **Все wake строго изолированы от Telegram-MCP** (щит от `AUTH_KEY_DUPLICATED`) + потолки (max-turns, budget-usd, wakes-per-hour, дневной cap) + зажатый permission-mode.
5. **Состояние в файлах** (mailbox + seen-keys + heartbeat + лог) → события между тиками не теряются, переживают сон/рестарт машины.
6. **Расширение = строка в `signals.yaml`**, а не новый демон.

## Живой тест поймал собственный баг
Первый прогон: сторож будил сам себя, потому что его файлы состояния лежали внутри той же папки, которую он сторожил → вынесены в отдельную `_state/`. Позже добавлен **owner-machine guard в коде**: только машина-хаб реально тикает, тик на не-той машине = no-op — класс «две машины тикают одновременно» закрыт на уровне кода.

## Guard-таблица (страховки)
| отказ | guard |
|---|---|
| duplicate firing | стабильный ключ + ledger seen-set + hub-only запуск |
| overlap | lockfile / mutex |
| missed event | mailbox-файл ждёт; SessionStart дренирует очередь |
| silent failure | heartbeat каждый тик + dead-man «beat старше 2× интервала → алерт» |
| token/$ runaway | default ping (0 ток.) + бюджет-потолки + cooldown |
| prompt-injection в сигнале | wake-промпт не доверяет содержимому файла как инструкции; tools зажаты |
