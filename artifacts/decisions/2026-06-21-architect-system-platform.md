# Decision Memo — Архитектор системы (AK-47-наблюдаемость)

> Итог Alpha Protocol 2026-06-21. Триггер: внешний DR-отчёт + запрос «нужен архитектор, документация и покрытие тестами всей системы».

## Проблема
Чем больше работы в Claude Code + волте (скиллы, скрипты, ночные роботы, миграция между машинами), тем больше «империя» знаний и автоматизаций. Нет единой карты, нет статуса «что работает / что тихо отвалилось», нет систематических тестов. Нужен **архитектор**, который держит карту системы и решает, где нужны проверки.

## Что знаем (Recall)
Ключевой вывод: **система НЕ greenfield — ~80% слоёв уже построено, просто россыпью и без сшитой карты.** Уже было: ручной инвентарь автоматизаций (разъезжался), health-дашборды, session-monitor, 3-2-1 бэкапы, десятки decision-заметок, ~44 скилла, ~190 скриптов. Дыры: единого запросимого каталога ВСЕГО нет; графа зависимостей нет; единого health-экрана нет; восстановление из бэкапа ни разу не проверено.

## Deep Research (отчёт на руках)
Скелет верный: **каталог-CMDB + граф зависимостей + SRE-наблюдаемость + ночной «архитектор»-сверщик + тесты по уровням важности + deterministic-first** (LLM только предлагает, код проверяет). НО конкретный стек отчёта (Prometheus / Grafana / Neo4j / Kopia / Backstage / MkDocs / GitHub Actions) = enterprise-обвес для команды → нарушает AK-47 (не-инженер сам не починит). **Отвергнут.**

## Решение — Вариант B «Архитектор по AK-47»
Тот же скелет DR, но на существующих кирпичах. Перевод enterprise → простой стек:
- Каталог: **один SQLite** (`system.db`) + детерминированный сканер. ~~Backstage~~
- Граф: **NetworkX-lite в Python** (рёбра из текстовых ссылок). ~~Neo4j~~
- Документация: **markdown прямо в волт**. ~~MkDocs~~
- Статус: **один статичный HTML** + health-score. ~~Prometheus/Grafana~~
- Бэкап: существующий 3-2-1 + **ежемесячный restore-drill**. ~~Kopia/restic~~
- Оркестрация: **планировщик ОС**. ~~GitHub Actions/systemd~~
- Архитектор: **детерминированный ночной прогон (0 токенов)**; LLM-предложения опциональны.

## Что построено (всё работает)
1. Сканер → каталог **829 активов** (скиллы, скрипты, задачи, MCP, БД, дашборды, заметки-память, decision-заметки, MOC, машины, цели бэкапа) + тонкий overlay (integrity-tier / owner / lifecycle).
2. Health-дашборд + живой авто-инвентарь. Health-score = 35% integrity · 25% recover · 20% freshness · 10% docs · 10% ops.
3. Граф зависимостей (скилл→пайплайн/БД, задача→скрипт) → детектор «скрипты без робота и скилла» (кандидаты в мёртвый код) и «пайплайны без расписания».
4. Тесты по уровням: nightly (свежесть каталога, битые активы), daily (свежесть бэкапа), weekly (integrity БД), monthly (**restore-drill** — реально вытащил файл из истории, доказал восстановимость).
5. Ночной оркестратор (скан → доки → тесты → бэкап, 0 токенов) + RED-пинг только на critical-падения (SRE: будим только на симптом) + read-only скилл `/arch`.

**Архитектор окупился сразу:** (а) нашёл ~10 «невидимых» ночных роботов, о которых ручной инвентарь не знал (ОС крутила ~32 задачи против ~20 в списке); (б) поймал реально падающую ночную задачу, которую инвентарь считал здоровой — оказалось benign lock-коллизией, но флаг подсветил → расследовали → правильный фикс.

## Урок
Метрика покрытия должна **измерять факт, а не выдумывать вердикт**: позже панель ложно кричала «offsite-бэкапа нет», хотя бэкап был — счётчик возвращал захардкоженный «warn» вместо реальной проверки. `result != 0` у задачи ≠ всегда «сломано».

## Риски / откат
Главный риск — переусложнить самого архитектора → лечится AK-47: 1 БД + 3 скрипта + 1 HTML + 1 задача, 0 сервисов, 0 токенов. Архитектор каталогизирует и себя (его freshness = метрика на дашборде).
