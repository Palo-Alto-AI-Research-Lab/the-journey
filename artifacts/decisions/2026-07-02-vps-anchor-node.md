# Decision Memo — VPS «якорь» (по Alpha Protocol R+DR)

> Публичная версия — конкретные адреса, аккаунты и креды инфраструктуры опущены.

## Проблема
Всё живое (рутины, шина между машинами, watchdog'и, approval-relay, кворум) привязано к дому. Дом умер (свет / интернет / пожар) → данные целы (3-2-1-бэкап есть), но система немая. Нужен always-on узел вне дома. Плюс мандат: узел = **второй свидетель кворума** флота.

## Текущие знания (recall)
- Волт: сотни тысяч мелких файлов, Syncthing-звезда из 4+ машин; ночной git-bundle → облако + локально (3-2-1 живой, healthcheck еженедельный).
- Уже есть боевые VPS под другие задачи (с чувствительными данными) — новый узел НЕ селим на них (blast-radius).
- GPU-работа и авторский голос остаются на домашнем хабе; VPS = CPU-only always-on.
- Гардрейлы: агент без разрушительного доступа к единственной копии; канон НЕ на VPS; Tier-2 через QQQ-approval.

## Результаты Deep Research (три независимых трека сошлись)
1. **Роль = узкий control-plane** («наблюдать, ставить в очередь, ретраить, уведомлять» — НЕ уничтожать / тратить / публиковать). Не второй хаб.
2. **Новый отдельный VPS** — не на боевые хосты с лидами/деньгами.
3. **Полной реплики волта НЕТ на старте.** Сотни тысяч мелких файлов = worst-case для синка (первичная синхронизация может тянуться днями). Паттерн: защищённый доступ к живому хабу + **маленькая отдельная шара** (ровно шина для witness-роли). Полная реплика — позже и только по доказанной нужде (receive-only curated subset).
4. **Control-plane должен быть LLM-free:** кворум / heartbeat / шина НИКОГДА не зависят от LLM — обычный Python + systemd/cron. Claude Code на якоре = «удобство, не SLA». Порядок стройки: witness → Telegram → Claude (последним).
5. **Claude Code на подписке = best-effort лента, не прод-инфраструктура:** долгоживущий OAuth-токен для скриптов, 1 headless-задача за раз, wrapper (лог + fail-closed на лимитах + идемпотентность), свой cron/systemd. Станет mission-critical → отдельный billing для unattended-части.
6. **Telegram-сессия рождается НА VPS** и никогда не мигрирует между машинами (иначе `AUTH_KEY_DUPLICATED`); только read-mostly / low-rate / approval-gated на зрелом аккаунте с 2FA; hard-stop на отзыве сессии → «human-only» режим + алерт. Первые/массовые сессии — не с датацентр-IP.
7. **Бэкап: Restic ДОПОЛНЯЕТ git-bundle** (bundle = git-история; restic = состояние машины: конфиги, сессии, очереди). Ежемесячный `restic check`, квартальный restore-drill на throwaway. Append-only.
8. **Гигиена хостинга:** 2FA, чистая регистрация, verified billing, runbook «хостинг недоступен»; узел обязан пересобираться из IaC + бэкапа (данные после lock не гарантированы).

## ⭐ Обновление 2026-07-03 — куплено + смена роли
Сервер заказан и поднят. **Осознанные отклонения от плана (решение Антона):**
1. **Роль:** из «узкий control-plane witness» → **рабочая машина под Claude Code-сессии** (нужна память под 5–15 сессий). Это тот самый «role creep», что мемо запрещало — принято сознательно как новая цель узла.
2. **Железо:** планировавшийся дешёвый ARM был распродан во всех EU-локациях (дефицит) → взяли доступный x86 втрое дороже плана (цена срочности).
3. Больше RAM сразу вместо старта на минимуме.
- **PENDING (Phase 0 хардунг):** первый вход, SSH-ключ, non-root юзеры, файрвол + защищённая сеть, потом Claude Code.

## Риски
Token expiry / refresh races (норма эксплуатации — ловить wrapper'ом) · отзыв TG-сессии · **«role creep»** (самый глубокий: якорь за полгода вбирает канон/секреты/CRM → становится той самой единой точкой отказа, которую мемо запрещало) · рост логов · lockout хостинга.

## Рекомендация — поэтапный rollout (GO после «+»; закупка = Tier-2)
Phase 0 закупка+харднинг → Phase 1 control-plane + witness (ДО всякого AI; acceptance-тест: **выключить хаб → VPS шлёт heartbeat-loss алерты и принимает approvals**) → Phase 2 Claude Code guarded → Phase 3 TG read-only → Phase 4 (опц.) micro-folder волта → Phase 5 Restic + retention + restore-drill.
